Introduction &amp; Project Overview1.1  BackgroundPhenikaa University Library manages the borrowing and returning of academic documents for thousands of students, lecturers, and researchers every semester. The traditional paper-based workflow — requiring physical card presentation, handwritten loan slips, and manual ledger entries — is time-consuming, error-prone, and difficult to audit.This project digitises that workflow by implementing a full-stack web application that enforces the official Phenikaa University Library Borrowing Regulation (Document Code: QT.03.TV.DV, issued under Decision No. 688/QD-DHP-DT&amp;QLSV, 27 December 2020). Every business rule in the regulation — borrowing quotas, loan durations, renewal limits, and late fines — is embedded directly into the application logic.1.2  Project ObjectivesFully digitise the 8-step document borrowing process defined in QT.03.TV.DVEnforce role-based borrowing rules for Students, Lecturers, and ResearchersProvide a cart-based self-service interface for readers to request loans onlineProvide an administrative interface for librarians to approve, reject, and process returnsAutomatically calculate overdue fines (2,000 VND per day) on book returnRestrict restricted documents (RESTRICTED type) from home borrowing1.3  Technology StackLayerTechnologyVersion / NotesBackend FrameworkSpring Boot3.3.5 — Java 21, MavenWeb MVCSpring Web MVC + ThymeleafServer-side template renderingSecuritySpring SecurityForm-based login, role-based accessORM / PersistenceSpring Data JPA / HibernateH2 in-memory (dev), MySQL-compatibleDatabase (Dev)H2 In-MemoryAuto-seeded via data.sql on startupDatabase (Prod)MySQLSwitch via application.propertiesValidationJakarta Bean ValidationConstraint annotations on DTOsUI FrameworkBootstrap 5.3.2Responsive layoutIconsBootstrap Icons 1.11.3Vector icon libraryBuild ToolMaven 3.8+pom.xml dependency managementLanguageJava 21Records, sealed classes, pattern matching2. System Architecture4.1  Architectural Pattern — MVCThe application follows the Model-View-Controller (MVC) architectural pattern, implemented through Spring Boot's layered conventions:LayerPackageResponsibilityKey ClassesPresentationcontroller/Handle HTTP requests, bind form data, redirectClientController, AdminController, AuthControllerBusiness Logicservice/Enforce business rules, orchestrate transactionsLoanService, BookService, UserServiceRule Engineutil/Encapsulate all borrowing rule calculationsRuleEngineData Accessrepository/JPQL queries, Spring Data interfacesLoanRepository, BookRepository, LoanDetailRepository, ...Domain Modelentity/JPA-mapped domain entities and enumerationsLoan, LoanDetail, Book, User, BorrowingRuleDTOsdto/Decouple form/session data from domain objectsCartItem, BookForm, ValidationResult, ReturnBookFormViewtemplates/Thymeleaf HTML templates rendered server-sideclient/, admin/, auth/, layout/4.2  Security ArchitectureSpring Security is configured with the following access control rules:/admin/**  —  requires ADMIN role/client/** —  requires any authenticated user (STUDENT, LECTURER, RESEARCHER, ADMIN)/login     —  public access/h2-console/** — developer access only (web-allow-others=false)Authentication uses Spring Security's form-based login backed by CustomUserDetailsService, which loads users by student_code from the database. Passwords are stored as BCrypt hashes.4.3  Session ManagementThe shopping cart is stored in the HTTP session (HttpSession) as a List&lt;CartItem&gt;. The session timeout is configured to 60 minutes. Cart data is cleared upon successful loan submission.4.4  Transaction ManagementAll mutating service methods are annotated with @Transactional. This ensures that multi-step operations such as loan submission (create Loan + create LoanDetails + decrement available_copies for each book) are atomic — either all steps succeed or all are rolled back.4.5  Request Flow DiagramA typical loan submission follows this path:Browser  →  POST /client/loans/submit         →  ClientController.submitLoan()         →  LoanService.submitLoan(user, cart)               ├─ RuleEngine.validate(user, cart)   [3-layer check]               ├─ loanRepository.save(loan)         [PENDING]               ├─ for each CartItem:               │     loanDetailRepository.save(detail)               │     book.availableCopies -= 1               └─ return ValidationResult.ok()         →  session.removeAttribute("cart")         →  redirect: /client/my-loans5. Conclusion &amp; Future Work10.1  AchievementsThis project successfully delivers a fully functional Library Management System that:Implements all 18 features covering the complete lifecycle of a document loan — from catalogue browsing to overdue fine collection.Faithfully enforces the official Phenikaa University borrowing regulation (QT.03.TV.DV) with 12 borrowing rules across 3 reader types and 4 document types.Follows clean Spring Boot MVC architecture with clear separation of concerns: no business logic in controllers, all rules encapsulated in RuleEngine.Identified and fixed 3 bugs in the original implementation: a critical data-loss bug in loan cancellation (LazyInitializationException + missing save), a security vulnerability allowing any user to renew any loan, and a logic error permitting renewal of already-returned items.Provides a safe development environment using H2 in-memory database with automatic seed data, and a clear migration path to MySQL for production.10.2
